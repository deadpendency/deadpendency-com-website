Why Use Deadpendency?

Updating dependencies is awesome

Minimal effort, large value

With a few keystrokes a developer can bump an application dependency version. For this effort, they (likely) improve the quality of their software.

In some cases dependencies have breaking changes and more effort is required to upgrade. However, these bigger upgrades typically provide greater value (ie. a simpler, easy to use API). In this case updating a dependency is still usually an excellent pay off for the time investment.

Dependency updates are mandatory

In the modern world of software development, there is constant platform churn. Most major programming languages release major updates every year. Beyond programming languages, there is an array of platforms modern software depends on. There are also inter-dependencies between an application's dependencies, making freezing any one to a particular version difficult.

The question is how long can dependency updates be delayed before it becomes a problem. Once significantly delayed, updating can becoming much more difficult and time consuming. In some cases if left too long an application can become undeployable without doing these updates, as dependent platforms have been discontinued.

Amazing dependency update tools

Dependency updates are important, but can be time consuming keeping up to date with many dependencies.

Thankfully there are excellent tools for keeping dependencies up to date such as Dependabot.

Dependency update tools have a gap

Open source projects often become inactive

In open source, projects often become inactive. Usually development is ceased and they sit bit rotting, or might slowly trickle along, but will never get new features. In some cases the dependency will be explicitly discontinued.

This isn’t necessarily a bad thing. Sometimes a library explores a new approach, the community learns from it, but ultimately this was not the end game solution. A new library can be developed with the better hindsight and the cycle continues.

Automated dependency tools ignore activity

Dependencies that are dead or are on life support will either never have a new version released, or very infrequently. This is the gap in the automated dependency update approach as these tools do not alert you of such a problem.

When you depend upon an inactive dependency, you are getting all the same downsides of never updating a dependency. Thus, to be in the best possible spot around dependencies, you need know your dependencies are actively being updated and released and be on the latest version.

This is not a flaw in automated dependency updating, which is excellent value to set up. It just means a different but related tool is needed like Deadpendency for maximal dependency health.

Deadpendency to the rescue

Constant safeguarding against unhealthy dependencies

Deadpendency is a GitHub app that runs a check against every push of code. This means over time, as maintenance works is done there are constant checks of dependency health. Continual checking here is important as dependencies that are healthy at a given point in time can be discontinued in the future.

Should I stop using 'dead' dependencies?

Inactive dependencies will bit rot and fall behind with the platform churn of modern software development. They are 'not optimal' to use.

However, sometimes you don't have a choice. There may be no alternative, or active alternatives have key flaws for your use case. Deadpendency doesn’t imply you should never use dependencies that are 'dead'. Rather, it is there to make the choice a conscious and deliberate one.

Of course, in some cases there is a healthy alternative and indeed it should be switched to. This is the case where Deadpendency provides the most value.

Other benefits

Protection against depending on forks

A common issue with using open source dependencies is dealing with bugs. In this case often the dependency will be forked and the fork used as a git dependency for a quick fix. Meanwhile, the fix is proposed via PR to the upstream repository. This can involve a lot of back and forth and delay before being merged and then released in a new version.

In this case dependency update tools (most of them at least) will then ignore new versions of the dependency, as it must assume the git dependency is still required. This means the developers need to remember to come back and un-depend on the fork. With Deadpendency the dependency immediately gets a warning as a forked repo. If overtime, it is left in place, the fork will not be getting commits and thus gets flagged as inactive.

An inactive dependency fails the check, so developers will be clearly reminded to revisit this fork usage.

A snapshot of dependency health

Deadpendency flags dependencies that are quite unhealthy. It also flags dependencies with warnings that are starting to become unhealthy. This means at any time the Deadpendency report can be reviewed for a quick snapshot of dependency health.

The Deadpendency report includes links to the dependency package and source repository enabling further review.

A gatekeeper for healthy dependencies

Much like a code linting check, Deadpendency acts as a fast feedback mechanism when unhealthy dependencies are attempted to be added via a PR. This works because Deadpendency runs against an exact commit sha.

In this way, Deadpendency can save developers time by blocking unhealthy dependencies. Or at least, prompting a justification to be included for their usage.
